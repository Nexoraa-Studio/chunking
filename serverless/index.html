What's happening under the hood

Cold Fargate boot is ~60–90 s on the first job (ECR pull). Warm boot is ~15–30 s while the image is cached. The actual pymupdf+DP pipeline is ~5–15 s for a typical 20–50 page PDF. Status is polled every 3 s from the job's status.json in S3.

If something goes wrong the task writes an error traceback into status.json and you'll see it on the "Job details" tab.

Architecture

Browser → API Gateway HTTP API → Lambda chunking-api (thin API).

Upload bypasses Lambda: the browser POSTs directly to S3 chunking-jobs-…apsouth1 using a presigned URL the Lambda hands back.

On Run, Lambda calls ecs:RunTask on task-def chunking-worker:1 (1 vCPU / 4 GB Fargate). The task reads the PDF from S3, runs pymupdf → structural → coarse-DP, uploads chunks.zip + status.json back to the same S3 prefix, then exits.

Download is a 302 redirect to a presigned GET URL on S3.

Why API Gateway instead of Lambda Function URL?

The Nexoraa org SCP blocks Lambda Function URLs with AuthType=NONE. IAM-signed invoke works (we tested), but a browser can't sign AWS requests without leaking credentials. API Gateway sits on the outside of that SCP, so it becomes the public front door. A Function URL still exists on the Lambda at xrsklj3f2mezkfiigs7nmp5pm40zbprf.lambda-url.ap-south-1.on.aws but it's locked to AWS_IAM.

What this UI does NOT support (and why)

Live log stream: API Gateway HTTP API doesn't support server-sent events cleanly; would need a WebSocket API or CloudWatch Logs tail. Polling status.json every 3 s is good enough for the current job's fidelity.

In-browser chunk viewer: would need Lambda endpoints that read the chunks.zip contents from S3. Can be added — the ECS-Express stack has it.
